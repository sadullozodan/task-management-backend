## Decisions Made

- **Many-to-many via `ModuleIssue` join table** (unlike Cycles which uses a `cycle_id` FK on Issue): an issue can belong to multiple modules simultaneously, matching the Plane data model from TZ §5.13.
- **`status` is stored explicitly** (not date-derived like Cycles): modules are thematic groupings, not time-boxed sprints, so status is a deliberate user choice.
- **`addIssuesToModule` uses `createMany + skipDuplicates`**: idempotent — re-adding an issue that's already in the module is a silent no-op; `added` count reflects only genuinely new rows.
- **Lead validation**: when `lead_id` is set on create or update, the service verifies the user is a member of the project (not just the workspace) to prevent assigning an external user as lead.
